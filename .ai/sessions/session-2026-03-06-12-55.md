### Phase 2 — 핵심 구현

#### Task 1+2: 5단계 리스크 레벨 + _check_intraday_risk()
- `_calc_intraday_multiplier()`: 순수 룰베이스 레벨 판정
  - NORMAL(1.0): KOSPI > -1%, VIX < 25
  - CAUTION(0.8): KOSPI <= -1% OR VIX >= 30
  - WARNING(0.6): KOSPI <= -2% OR VIX >= 35
  - DANGER(0.3): KOSPI <= -3% OR VIX >= 40
  - CRITICAL(0.0): KOSPI <= -4%
- `_check_intraday_risk()`: 기존 `_check_circuit_breaker()` 완전 대체
- `min(council_multiplier, intraday_multiplier)` 역전 방지
- WARNING 이상 → regime downgrade (BEAR/STRONG_BEAR)

#### Task 3: TradingContext 모델 확장
- `intraday_risk_level: str = "NORMAL"`
- `intraday_multiplier_raw: float = 1.0`
- `council_multiplier_raw: float = 1.0`
- Council 실행 시 intraday 상태 초기화 + recovery key 클리어

#### Task 4: DAG interval 5분
- `enhanced_macro_quick`: `30 9-14` → `*/5 9-15`, `max_active_runs=1`
- Airflow reserialize로 즉시 적용 확인

#### Task 5: 텔레그램 알림
- `_send_risk_alert()`: 레벨 변경 시에만 발송 (중복 없음)
- 악화: ⚠️ 아이콘, 개선: ✅ 아이콘, multiplier min 적용 표시

#### Task 6: 회복 로직
- `_apply_recovery_logic()`: 악화 즉시, 회복 단계적
  - CRITICAL: 6사이클(30분) → DANGER
  - DANGER: 4사이클(20분) → WARNING
  - WARNING: 2사이클(10분) → CAUTION
  - CAUTION: 1사이클(5분) → (최소 CAUTION, NORMAL은 Council만)
- Redis `intraday:risk:recovery_start` 타이머 기반

#### Task 7: 구조화 로깅
- binding (council/intraday) 식별 포함 INFO 로그
